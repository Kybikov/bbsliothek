н# BBSliothek — Повний опис проекту (для себе, українською)

---

## Що це взагалі за проект

Це шкільний Abschlussprojekt для Lernfeld 8 ("Daten systemübergreifend bereitstellen").
Задача — зробити програму для зберігання навчальних матеріалів (PDF, docx, py, jpg тощо)
з базою даних MySQL і інтерфейсом на Python.

Ми зробили **десктопний додаток** на Python із красивим UI через бібліотеку **Flet**
(Flutter-based), який підключається до **MySQL** бази даних через **mysql-connector-python**.

## База даних — таблиці та їх зміст

### Таблиця `rollen` (ролі)
Дві ролі: Lehrkraft (вчитель) і Auszubildende (учень/студент).
Проста таблиця, потрібна щоб знати хто є хто.

### Таблиця `benutzer` (користувачі)
Юзери системи. Кожен юзер має роль (FK на rollen).
Поля: id, роль, відображуване ім'я, email, дата реєстрації.
У нас немає реального логіну — при завантаженні просто вибираєш юзера зі списку.

### Таблиця `themengebiete` (теми)
Теми матеріалів: Informatik, Mathematik, Pflege, Wirtschaft.
Кожен матеріал належить рівно до однієї теми.

### Таблиця `materialien` (матеріали)
Головна таблиця. Кожен матеріал — це заголовок + тема + автор.
Важливе поле: `aktuelle_version_id` — вказує на поточну версію файлу.
Це дозволяє мати кілька версій одного матеріалу.

### Таблиця `material_versionen` (версії)
Кожен upload — це нова версія. Тут зберігається:
- `versionsnummer` — номер версії (1, 2, 3...)
- `dateiname`, `dateityp`, `dateigroesse_bytes` — мета-дані файлу
- `speicherstrategie` — DB_BLOB або DATEISYSTEM
- `blob_inhalt` — бінарний вміст файлу (якщо < 1MB)
- `dateipfad` — шлях до файлу (якщо >= 1MB)
- `checksumme_sha256` — хеш файлу для перевірки цілісності

### Таблиця `kommentare` (коментарі)
Коментарі до матеріалів. FK на materialien і benutzer.
Поля: текст, автор, дата створення, дата останньої зміни.

### View `vw_material_aktuell`
Це не таблиця, а "віртуальна таблиця" — SQL VIEW.
Об'єднує materialien + material_versionen + themengebiete + benutzer в одному SELECT.
Щоразу коли треба показати список матеріалів — використовуємо цей view.
Це зручно: замість великого JOIN в кожній функції — просто SELECT * FROM vw_material_aktuell.

## Функціонал додатку — сторінка за сторінкою

### 1. Materialien (список матеріалів)
- Показує всі матеріали з бази у вигляді таблиці
- Колонки: ID, Titel, Typ, Thema, Autor, Version, Größe, кнопка завантаження
- Фільтрація за назвою (LIKE пошук) і типом файлу
- Кнопка "Herunterladen" — зберігає файл в папку `downloads/` і відкриває його

**Як завантаження працює:**
- Якщо стратегія DB_BLOB → читає бінарні дані з бази і записує у файл
- Якщо стратегія DATEISYSTEM → просто копіює файл з `storage/materials/`

### 2. Upload (завантаження нового матеріалу)
- Поле для шляху до файлу (треба вписати вручну або вставити з буферу)
- Dropdown для вибору автора
- Dropdown для вибору теми
- Поле для назви матеріалу
- Опційно: поле Material-ID для нової версії існуючого матеріалу

**Логіка стратегії зберігання:**
```
файл < 1MB  →  читає весь файл в пам'ять → зберігає як BLOB в MySQL
файл >= 1MB →  копіює файл в storage/materials/ → в БД зберігає тільки шлях
```

**Чому так:** зберігати великі файли в БД погано для продуктивності.
MySQL погано справляється з багатьма великими BLOB-ами.

### 3. Abfragen (стандартні запити)
7 запитів кліком: вибираєш зліва → результат показується справа у таблиці.

Запити по типах (як вимагала задача):
- Запити 1, 2 → Aggregation (COUNT, AVG, GROUP BY)
- Запити 3, 4 → Inner Join (2 таблиці)
- Запит 5 → Join + Aggregation
- Запити 6, 7 → 2+ Inner Join через кілька таблиць

### 4. Kommentare (коментарі)
- Вибираєш матеріал → відразу підвантажуються всі коментарі
- Форма внизу для нового коментаря: вибір автора + текст
- Коментарі показуються як картки з ім'ям автора і датою

### 5. Themen (теми)
- Список існуючих тем зліва
- Форма для додавання нової теми справа

## Де відхилилися від задачі

### Краще зробили ніж вимагалось:

**1. Flet UI замість консолі**
Задача каже "Eingabemaske in Textform" (тобто консоль).
Ми домовились з вчителем і зробили красивий десктопний UI через Flet.
Плюс: виглядає набагато краще, легше демонструвати.
Мінус: технічно відступ від ТЗ, але погоджено.

**2. Версіонування файлів**
Задача вимагає upload і download. Ми додали версіонування:
кожен новий upload створює нову версію, стара зберігається.
Це не вимагалось але логічно і корисно.

**3. SHA-256 checksums**
Для кожного файлу рахується хеш для перевірки цілісності.
Задача цього не вимагала, але це "правильно" з точки зору розробки.

**4. SQL VIEW**
Зробили `vw_material_aktuell` — view яка об'єднує всі таблиці.
Задача цього не вимагала, але спрощує всі запити.

**5. Індекси в БД**
Додали індекси на FK-колонки для швидшого пошуку.
Задача не вимагала, але це хороша практика.

### Гірше зробили або не зробили:

**1. Немає реального логіну**
Задача не вимагала, але логічно що в системі для вчителів і учнів
має бути аутентифікація. У нас просто dropdown "хто ти?".
TODO коментар в коді є, але реалізації немає.

**2. FilePicker не спрацював**
Спочатку планували красиве вікно вибору файлу (ft.FilePicker).
Через баги Flet 0.80 довелось замінити на ручне поле для шляху.
Тепер треба самому вписувати або вставляти шлях — не дуже зручно.

**3. Немає редагування коментарів**
Коментарі можна тільки додавати, але не редагувати і не видаляти.
Таблиця має поле `geaendert_am` (дата зміни) — значить планували,
але в UI не реалізували.

**4. Немає пошуку за автором в UI**
В `datenbank.py` функція `materialien_laden()` підтримує фільтр за `autor_id`,
але в UI цього фільтру немає на сторінці Materialien.
Просто не встигли/забули додати dropdown для автора.

**5. `os.startfile()` тільки Windows**
Автоматичне відкриття файлу після завантаження працює тільки на Windows.
На Linux/Mac треба `subprocess.run(["xdg-open", pfad])` — є коментар в коді,
але не реалізовано умовно (if sys.platform == ...).

Могли зробити `.env` файл і `python-dotenv` — але це додаткова залежність

В `datenbank.py` всі функції окремі, кожна сама відкриває і закриває з'єднання.

## Технічні деталі які варто знати для презентації

### Чому mysql-connector-python, а не SQLAlchemy?
SQLAlchemy — це ORM, більш потужний але складніший.
Для студентського проекту простіше показати що ти розумієш SQL напряму.
mysql-connector-python = "чистий" MySQL без прошарків.

### Чому Flet, а не Tkinter або PyQt?
- Tkinter — стандартний Python UI, але виглядає як 1995 рік
- PyQt — потужний але складний, потребує окремої ліцензії для комерційного використання
- Flet — будується на Flutter, виглядає сучасно, код простий, cross-platform

### Чому BLOB для малих файлів, а не завжди файлова система?
- Зберігання в БД: транзакції, backup разом з даними, простота
- Файлова система: краща продуктивність для великих файлів
- Межа 1MB — компроміс між зручністю і продуктивністю

### Що таке транзакція в upload?
В `material_hochladen()` є `conn.start_transaction()` і `conn.commit()`.
Якщо щось піде не так посередині (наприклад не вдалось INSERT версію) —
`conn.rollback()` відміняє всі зміни. Або все зберігається, або нічого.
Це важливо щоб не мати "половинчастих" записів в БД.

## Баги які виникали під час розробки (для презентації)

---

### FilePicker — повна історія (найцікавіший баг)

Це найдовша і найцікавіша проблема в проекті. Тривала кілька ітерацій виправлень.

**Мета:** зробити кнопку "Datei auswählen" яка відкриває системне вікно вибору файлу.

---

#### Спроба 1 — по старій документації

Знайшов туторіали де написано передати `on_result` в конструктор:

```python
file_picker = ft.FilePicker(on_result=datei_ausgewaehlt)
page.overlay.append(file_picker)
```

**Результат:** `TypeError: FilePicker.__init__() got an unexpected keyword argument 'on_result'`

Конструктор більше не приймає `on_result`. API змінився в Flet 0.80.

---

#### Спроба 2 — присвоїти після створення

Прочитав що `on_result` треба задавати як property після створення:

```python
file_picker = ft.FilePicker()
file_picker.on_result = datei_ausgewaehlt  # задаємо після
page.overlay.append(file_picker)
```

**Результат:** `Unknown control: FilePicker`

Flet 0.80 не розпізнає FilePicker як валідний control взагалі, якщо додавати його динамічно при переході на сторінку Upload.

---

#### Спроба 3 — ініціалізація на старті

Прочитав що FilePicker треба додавати до `page.overlay` **один раз при старті**, а не динамічно:

```python
def main(page):
    file_picker = ft.FilePicker()
    page.overlay.append(file_picker)  # на самому початку main()
    # ... далі весь код
```

**Результат:** `Unknown control: FilePicker` — тепер при відкритті Login-сторінки, ще до будь-якої навігації.

Виявилось що FilePicker в Flet 0.80 **взагалі не підтримується через `page.overlay`** — ні динамічно, ні на старті.

---

#### Спроба 4 — тимчасове рішення: TextField

Поки шукали правильне рішення — замінили FilePicker на просте текстове поле де юзер вводить шлях вручну. Зробили коментар в коді і README що це тимчасово.

```python
pfad_feld = ft.TextField(
    label="Pfad zur Datei",
    hint_text=r"z.B. C:\Users\...\datei.pdf",
)
```

Програма запрацювала. Але "Datei auswählen" кнопки не було.

---

#### Спроба 5 — знайшли правильну документацію

Знайшли офіційну галерею прикладів Flet: https://flet.app/gallery/utility/filepicker

Там три варіанти. Прочитали код і зрозуміли: **в Flet 0.80 FilePicker став асинхронним**.

Нова API:
```python
async def datei_auswaehlen(e):
    picker = ft.FilePicker()          # створюємо прямо тут
    files = await picker.pick_files() # async! без page.overlay!
    if files:
        pfad_feld.value = files[0].path
```

Ключові відмінності від старої API:
- `async/await` — виклик асинхронний
- Picker створюється **всередині** handler-функції, не глобально
- **Не треба** `page.overlay.append()`
- **Не треба** `on_result` callback

**Результат:** ✅ Працює на Desktop. Кнопка відкриває системне вікно вибору файлу, шлях підставляється автоматично.

---

#### Чому не працює в браузері?

В браузері `files[0].path` завжди `None`. Це **обмеження самого браузера** (не Flet) — з міркувань безпеки браузер не дає доступу до локального шляху файлу.

Для повного upload в браузері потрібен механізм `page.get_upload_url()` який вимагає `FLET_SECRET_KEY`. Ми вирішили не ускладнювати:
- **Desktop:** FilePicker працює повністю — шлях підставляється автоматично ✅
- **Браузер:** FilePicker відкривається, але шлях `None` → показується підказка "введіть шлях вручну" ⚠️

---

#### Підсумок FilePicker:

| Спроба | Код | Помилка |
|---|---|---|
| 1 | `ft.FilePicker(on_result=...)` | TypeError |
| 2 | `file_picker.on_result = ...` + overlay | Unknown control |
| 3 | overlay на старті | Unknown control при логіні |
| 4 | Замінили на TextField | Працює, але без діалогу |
| 5 | `async/await` без overlay | ✅ Працює |

**Для презентації:** це відмінний приклад того як одна проблема займає багато часу не через складність коду, а через зміни в API бібліотеки між версіями. Реальна розробка виглядає саме так.

---

### Інші баги (коротко)

1. **`ft.padding.only`** — видалено в Flet 0.80 → замінено на `ft.Padding(top=8, left=0, right=0, bottom=0)`
2. **`ft.border.all`** — видалено в Flet 0.80 → замінено на `ft.Border(top=ft.BorderSide(...), ...)`
3. **`ft.app(target=main)`** → `ft.run(main)` — функцію перейменували в 0.80
4. **`ft.run(target=main)`** → `ft.run(main)` — `target=` більше не keyword аргумент (помилка при "виправленні")
5. **`ft.ElevatedButton`** — deprecated → замінено на `ft.FilledButton`
6. **`page.window_width`** → `page.window.width` — window властивості перенесли в 0.80
7. **MySQL `errno 150`** — циклічний FK між `materialien` і `material_versionen` → вирішено через `ALTER TABLE` після обох `CREATE TABLE`
8. **`Unknown database 'bbsliothek'`** — база не існувала → треба запустити `01_schema.sql` спочатку
9. **`bytes()` при читанні BLOB** — MySQL Connector повертає `bytearray`, треба явно конвертувати в `bytes` при запису у файл (актуально для старої BLOB версії)

## Як запустити в браузері і на телефоні

### Браузер
```bash
python main.py --web
# Відкриється http://localhost:8550
```

### Телефон (через локальну мережу)
1. Запусти `python main.py --web`
2. Дізнайся IP комп'ютера: `ipconfig` (Windows) або `ifconfig` (Mac/Linux)
3. На телефоні відкрий браузер → `http://192.168.X.X:8550`
4. Телефон і ноутбук мають бути в одній WiFi-мережі

### Телефон (APK/IPA)
Теоретично можливо через `flet build apk`, але:
- Потрібен Android SDK / Xcode
- MySQL має бути доступний з телефону (не localhost)
- Для шкільного проекту — зайве

**Висновок:** для демонстрації достатньо web-режиму.

## Що показати на презентації

1. **Запустити програму** — показати навігацію, що вона взагалі працює
2. **Materialien** — показати таблицю, зробити пошук
3. **Upload** — завантажити реальний файл, показати що з'явився в таблиці
4. **Abfragen** — запустити 2-3 запити, пояснити типи (Aggregation, Join)
5. **Kommentare** — додати коментар до щойно завантаженого матеріалу
6. **phpMyAdmin** — показати саму БД, таблиці, записи
7. **Баги** — розповісти про Flet 0.80 API-зміни як приклад реального розвитку
